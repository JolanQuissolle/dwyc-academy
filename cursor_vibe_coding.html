Le paradigme Vibe Coding

Ce n'est pas "apprendre à coder avec de l'aide". C'est un nouveau mode de production : vous êtes l'architecte, l'IA est l'ingénieur. Comprendre la distinction change tout.

⚠️
L'erreur fatale du débutant : traiter ces outils comme des moteurs de recherche améliorés. Vous leur posez des questions. La réalité : ce sont des agents d'exécution. Vous leur donnez des missions, un contexte, des contraintes. La qualité de votre pensée détermine la qualité du résultat.

🧠 Le modèle mental

Vous n'écrivez plus de code. Vous spécifiez des comportements. La compétence qui compte n'est plus la syntaxe — c'est la capacité à décomposer un problème en tâches atomiques, à définir les contraintes avec précision, et à valider les outputs.

⚡ Le stack optimal

  • Bolt.new → Scaffold initial (15 min)
  • v0.dev → Composants UI complexes
  • Cursor → Dev quotidien, itération
  • Claude Code → Tâches lourdes autonomes

📐 Les 3 lois du vibe coding efficace

01

Contexte > Commande

L'IA produit en proportion directe du contexte que vous donnez. Un prompt avec architecture, stack, contraintes, et exemple produit 10x mieux qu'une commande vague. Investissez 2 minutes à contextualiser pour économiser 20 minutes de correction.

02

Décomposer avant d'exécuter

Ne demandez jamais "construis-moi une app". Découpez en micro-tâches : "crée le schéma de base de données", puis "génère l'API", puis "construis le composant UI". Chaque tâche doit tenir en une conversation.

03

Vérifier sans se noyer

Vous ne comprendrez pas tout le code généré. C'est normal et acceptable — à condition de tester le comportement, pas la syntaxe. Faites tourner, testez les edge cases, vérifiez la sécurité des points critiques (auth, données, paiement).

✦ .cursorrules — Le fichier qui change tout

Placé à la racine du projet, .cursorrules (ou .cursor/rules) injecte du contexte système dans chaque requête. C'est votre constitution de projet que l'IA respecte en permanence.

.cursorrules — Template production
# Stack & Architecture
STACK: Next.js 14 App Router, TypeScript strict, Tailwind CSS, Supabase
UI: shadcn/ui components uniquement, pas de bibliothèques tierces
STATE: Zustand pour l'état global, React Query pour les données serveur
AUTH: Supabase Auth — jamais stocker le token manuellement

# Conventions de code
STYLE: Composants fonctionnels uniquement, arrow functions
NAMING: PascalCase composants, camelCase hooks/utils, SCREAMING_SNAKE constants
IMPORTS: Utiliser les alias @/components, @/lib, @/hooks

# Règles de sécurité
SECURITY: Ne jamais exposer SUPABASE_SERVICE_ROLE_KEY côté client
VALIDATION: Zod pour toute validation d'input utilisateur
API: Toujours valider l'auth dans les Route Handlers

# Comportement attendu de l'IA
DO: Expliquer brièvement les choix techniques non-évidents
DO: Proposer le type TypeScript complet pour chaque fonction
DONT: Utiliser 'any' comme type TypeScript
DONT: Ajouter des dépendances npm non listées sans demander

🔬 Composer — Agent multi-fichiers

Le mode Composer (Cmd+I) permet à Cursor d'écrire, modifier et créer plusieurs fichiers en une seule conversation. C'est le mode à utiliser pour les features complètes.

Prompt Composer — Feature complète
TASK: Créer un système de commentaires pour les posts

CONTEXT:
- Table 'posts' existante avec colonnes: id, user_id, title, content
- Auth Supabase déjà configurée
- Composant PostCard.tsx existant à étendre

DELIVERABLES:
1. Migration SQL: table comments (id, post_id, user_id, content, created_at)
2. Policies RLS: lecture publique, écriture authentifiée, delete owner only
3. Hook useComments.ts avec fetch + optimistic insert
4. Composant CommentSection.tsx avec liste + formulaire
5. Intégration dans PostCard.tsx existant

CONSTRAINTS:
- shadcn/ui uniquement pour les composants UI
- Optimistic updates sur l'insert (UX fluide)
- Gestion d'erreur complète

🎯 @ References

Dans Chat et Composer, utilisez @ pour cibler précisément :

  • @fichier.ts — contexte d'un fichier
  • @dossier/ — tout un répertoire
  • @Codebase — recherche sémantique globale
  • @Web — recherche web en temps réel
  • @Docs — documentation indexée
  • @Git — diff ou commit spécifique

⚙️ Raccourcis essentiels

  • Cmd+K — Edit inline sur sélection
  • Cmd+L — Ouvrir Chat
  • Cmd+I — Ouvrir Composer
  • Cmd+Shift+L — Ajouter fichier au chat
  • Cmd+Enter — Envoyer avec agent mode
  • Escape — Annuler une suggestion Tab
  • Cmd+Z — Undo une modification IA

Module 02

Claude Code

Agent CLI d'Anthropic. S'exécute dans votre terminal, lit et écrit des fichiers, lance des commandes, git commit. L'autonomie maximale pour les tâches lourdes.

CLI Agent Multi-fichiers Git natif Autonomie CLAUDE.md

🚀 Installation & premiers pas

bash
# Installation
npm install -g @anthropic-ai/claude-code

# Lancer dans votre projet
cd mon-projet
claude

# Mode non-interactif (pour scripts/CI)
claude -p "Ajoute des tests pour toutes les fonctions dans src/utils/"

# Avec un fichier de contexte
claude --context CLAUDE.md

# Mode verbose (voir les actions en détail)
claude --verbose

📄 CLAUDE.md — La mémoire permanente

Equivalent du .cursorrules, mais pour Claude Code. Ce fichier est lu automatiquement à chaque session. Placez-le à la racine du projet.

CLAUDE.md — Template
# Contexte du projet
Application SaaS B2B de gestion de projets musicaux.
Stack: Next.js 14, TypeScript, Supabase, Stripe.

# Structure du projet
- src/app/          → Routes Next.js (App Router)
- src/components/   → Composants React réutilisables
- src/lib/          → Utilitaires et clients (supabase, stripe)
- src/hooks/        → Custom React hooks
- supabase/         → Migrations et Edge Functions

# Commandes importantes
- `npm run dev`         → Démarrer en local
- `npm run test`        → Tests unitaires (Vitest)
- `supabase db push`    → Appliquer les migrations
- `npm run typecheck`   → Vérifier les types TS

# Règles impératives
- TOUJOURS lancer les tests après modification
- TOUJOURS vérifier TypeScript avant de commit
- Ne JAMAIS push directement sur main
- Créer une branche feature/ pour chaque tâche

# Patterns à respecter
- Server Components par défaut, 'use client' uniquement si nécessaire
- Erreurs: Result pattern (pas de throw, retourner { data, error })
- Validation: Zod schemas dans src/lib/schemas/

🤖 Commandes agent — Ce que Claude Code peut faire

READ

Lecture de fichiers et navigation

Explore l'arborescence, lit les fichiers, comprend la structure. Demandez : "Explique-moi l'architecture de ce projet"

WRITE

Création et modification de fichiers

Crée, modifie, supprime des fichiers. Demandez : "Refactore tous les composants pour utiliser le nouveau design system"

EXEC

Exécution de commandes shell

Lance npm, git, supabase CLI, tests. Demandez : "Lance les tests, et si ils échouent corrige les erreurs"

GIT

Opérations Git complètes

Branch, commit, diff, merge. Demandez : "Crée une PR avec un changelog détaillé de ce que tu viens de faire"

✦ Prompts haute valeur pour Claude Code

Migration de codebase
Migre tous les composants de pages/ (Pages Router) vers app/ (App Router).
Respecte les conventions Next.js 14 : Server Components par défaut,
loading.tsx et error.tsx pour chaque route, metadata exports.
Lance le build à la fin pour vérifier qu'il n'y a pas d'erreurs.
Génération de tests
Analyse src/lib/utils.ts et src/hooks/.
Génère des tests Vitest avec couverture à 80%+ pour chaque fonction.
Utilise des mocks pour Supabase (pattern existant dans __tests__/setup.ts).
Lance les tests et corrige jusqu'à ce qu'ils passent.
Audit de sécurité
Fais un audit de sécurité du projet. Cherche :
1. Clés API exposées côté client
2. Routes API sans vérification d'auth
3. Queries sans RLS ou validation d'input
4. Dépendances avec vulnérabilités connues (npm audit)
Crée un rapport SECURITY_AUDIT.md avec les findings et les fixes prioritaires.
⚠️
Contrôle des permissions : Claude Code demande confirmation avant les actions destructives. Lisez attentivement ces confirmations. Utilisez claude --dangerously-skip-permissions uniquement dans des environnements isolés (Docker, CI) — jamais sur votre machine principale avec des données sensibles.

🗺️ Le Pipeline complet — De l'idée au prod

1

Spec & Architecture (30 min) — Claude.ai

Avant de toucher aux outils : utilisez Claude.ai (chat) pour affiner votre spec. Décrivez l'idée, demandez à l'IA de la questionner, d'identifier les edge cases, et de proposer l'architecture de données. Output : un document spec clair.

L'essentiel du travail. Ajoutez .cursorrules, ouvrez le projet, et utilisez Composer pour chaque feature. Chat pour les questions, Inline Edit pour les modifications précises.

Migrations, refactorings massifs, génération de tests, audits de sécurité, préparation de releases. Claude Code pendant que vous faites autre chose.

🚀 Techniques avancées

Chain-of-thought : Demandez "Réfléchis étape par étape avant de coder" pour les architectures complexes. L'IA planifie avant d'exécuter = moins d'erreurs.

Exemples positifs/négatifs : Montrez un exemple de ce que vous voulez ET un exemple de ce que vous ne voulez pas. Le contraste calibre précisément l'output.

Checkpoint explicite : "Avant de coder, liste les fichiers que tu vas modifier et dis-moi ton approche. J'approuve avant que tu procèdes." Évite les surprises.

Contrainte de taille : "La fonction doit faire moins de 30 lignes" ou "Le composant ne doit avoir qu'une seule responsabilité". Force l'IA à être chirurgicale.

✦
Axes de progression non demandés : (1) Maîtriser MCP (Model Context Protocol) dans Cursor — connecte l'IA directement à votre Supabase, GitHub, Figma. (2) Construire un workflow de génération d'images avec Replicate déclenché par Claude Code. (3) Apprendre à évaluer les outputs rapidement : lire les diffs git plutôt que le code complet, tester les interfaces plutôt que la logique ligne par ligne.
