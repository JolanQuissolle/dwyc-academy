n8n Mastery

Le cours le plus complet sur n8n — de l'installation à la construction d'agents IA autonomes. Tu vas comprendre chaque node, chaque expression, chaque pattern. Et construire de vraies automations qui tournent en production.

Qu'est-ce que n8n ?

n8n (prononcé "n-eight-n", pour "nodemation") est un outil d'automatisation de workflows open-source basé sur les nœuds. Pense à lui comme un Zapier/Make mais sans limites : tu peux auto-héberger, écrire du code custom, construire des agents IA, connecter n'importe quelle API — et tout ça visuellement.

Fondé en 2019, il compte aujourd'hui 66 000+ étoiles GitHub, 500+ intégrations natives, et est utilisé par des entreprises qui font tourner 200+ workflows critiques en production.

⚡
n8n vs Zapier vs Make : Zapier = simple mais limité et cher. Make = plus puissant mais interface complexe. n8n = le plus puissant, open-source, auto-hébergeable, avec du code natif, des agents IA LangChain, et une communauté technique. Coût : gratuit en self-hosted vs $49-599/mois pour les concurrents.

Installation & Setup

3 façons de lancer n8n. Le cloud pour commencer vite, Docker pour le pro, npm pour les tests locaux.

Option 1 — n8n Cloud (le plus simple)

Va sur n8n.io → "Start for free". Compte gratuit avec 2 500 exécutions/mois. Idéal pour tester et les projets légers. Pas besoin d'installer quoi que ce soit.

⚠️
Limite Cloud : Après 2 500 exécutions/mois, le plan commence à $20/mois. Pour une startup avec des workflows fréquents, le self-hosted Docker est 10x moins cher à terme.

Option 2 — Docker (recommandé pour la production)

docker-compose.yml

Interface & Canvas

Tout ce que tu dois savoir sur l'interface n8n pour travailler efficacement — canvas, exécutions, debug, raccourcis.

Anatomie de l'interface

📐
Canvas (zone de travail)
L'espace infini où tu construis tes workflows. Drag & drop pour déplacer les nodes. Scroll pour zoomer. Clic droit → menu contextuel.
▶️
Bouton Execute Workflow
Lance le workflow manuellement. Montre les données qui passent entre chaque node en temps réel. Indispensable pour débugger.
📋
Onglet Executions
Historique de tous les runs passés. Voir les succès, erreurs, durées. Rejouer une exécution. Inspecter les données de chaque step.
🔑
Credentials
Stockage sécurisé et chiffré de toutes tes clés API, tokens OAuth. Partagés entre workflows. Ne jamais hardcoder des secrets dans les nodes.
🏷️
Tags & Dossiers
Organiser tes workflows par projet, client, type. Tags colorés. Indispensable quand tu as 50+ workflows.
⚙️
Settings (workflow)
Error workflow, timezone, save on success/error, timeout. Configure par workflow, pas globalement.

Débugger un workflow

1
Exécute le workflow manuellement
Clique sur "Execute Workflow". Les nodes s'illuminent en vert (succès) ou rouge (erreur). Le nombre d'items qui passent est affiché sur chaque connexion.
2
Inspecte les outputs
Clique sur n'importe quel node après exécution → "Output". Tu vois les données JSON exactes qui sortent. C'est là que tu comprends ce qui se passe.
3
Rejoue depuis un node spécifique
Clic droit sur un node → "Execute Node". Lance uniquement ce node avec les dernières données reçues. Pas besoin de réexécuter tout le workflow.
4
Utilise le node Debug Helper
Ajoute un node "Set" après n'importe quel node pour logger les données actuelles. Ou connecte un node "Slack/Email" pour recevoir un dump des données en cas d'erreur.
💡
Astuce pro : Active "Always Output Data" sur les nodes qui peuvent ne rien retourner (ex: une query SQL qui revient vide). Sans ça, le workflow s'arrête silencieusement si un node ne produit aucun item — comportement contre-intuitif mais critique à connaître.

2 — Data Transformation (manipuler les données)

Ces nodes transforment les données sans appeler d'API externe. Ce sont les plus utilisés dans tout workflow sérieux.

✏️
Edit Fields (Set)
Le node le plus utilisé. Ajoute, modifie, supprime des champs dans les items. Renomme des clés. Crée de nouvelles valeurs avec des expressions. Le Set de toute transformation.
✂️
Filter
Supprime les items qui ne remplissent pas une condition. Ex: garder seulement les emails avec un montant > 1000€. Équivalent d'un WHERE en SQL.
📊
Aggregate
Combine plusieurs items en un. Somme, count, min, max, array. Ex: regrouper 50 transactions en un seul total.
✂️
Split Out
L'inverse de Aggregate. Transforme un item avec un array en N items séparés. Ex: une réponse API avec 10 objets → 10 items individuels.
🗑️
Remove Duplicates
Dédoublonne les items basé sur un ou plusieurs champs. Essentiel pour les syncs où des données peuvent arriver plusieurs fois.
🔢
Sort
Trie les items par n'importe quel champ, ascendant ou descendant, alphabétique ou numérique. Ou ordre aléatoire.
✂️
Limit
Garde seulement les N premiers items. Pour éviter de traiter 10 000 lignes quand tu veux juste les 100 premières.
📈
Summarize
Agrège des items comme un pivot Excel. Group by + sum/count/average. Crée des rapports sans code.

3 — Logic Nodes (contrôle du flux)

🔀
IF
Condition binaire. 2 sorties : "true" et "false". Ex: si statut = "actif" → branche A, sinon → branche B. Le plus simple à comprendre.
🔄
Switch
Condition multiple. Jusqu'à 10 sorties. Ex: si type = "lead" → CRM, si type = "client" → billing, sinon → spam. Équivalent d'un switch/case.
🔁
Loop Over Items
Itère sur des items en batch. Utile pour respecter les rate limits d'APIs (traiter 10 items, attendre 1 seconde, traiter les 10 suivants). Combiné avec Wait.
🔗
Merge
Fusionne plusieurs branches en une. Modes : Append (tous les items), Merge By Key (join SQL-like), Combine (zip), Choose Branch. Central dans les workflows parallèles.
⏸️
Wait
Pause le workflow. Attendre N secondes/minutes/heures. Ou attendre un webhook de validation externe. Ou attendre jusqu'à une date/heure précise.
📞
Execute Workflow
Appelle un autre workflow n8n. Permet la modularité : un workflow parent qui orchestre des sous-workflows spécialisés. Passe des données en input, récupère un output.
🔁
Stop and Error
Arrête le workflow et lève une erreur custom. Utile pour valider des conditions en amont et éviter des exécutions corrompues.
✅
No Op (No Operation)
Passe les données sans rien faire. Sert de "placeholder" ou de point de convergence visuelle dans le canvas pour garder un workflow lisible.

4 — Core Utility Nodes (indispensables)

💻
Code
JavaScript ou Python custom. Accès complet aux items via $input.all(). Peut importer des packages npm (en self-hosted). La soupape de sécurité quand aucun autre node ne suffit.
🌐
HTTP Request
Appel HTTP vers n'importe quelle API. GET/POST/PUT/PATCH/DELETE. Auth (API Key, Bearer, OAuth2, Basic). Pagination automatique. Le node qui connecte n8n à tout internet.
📁
Read/Write Files
Lire et écrire des fichiers sur le système (self-hosted). Travailler avec des PDFs, images, CSV. Convertir des formats binaires.
🔐
Crypto
Hash MD5/SHA256, signer des données, générer des UUIDs, encoder/décoder base64. Pour les intégrations qui nécessitent des signatures de requête (HMAC).
📅
Date & Time
Manipuler des dates : ajouter/soustraire, formatter, convertir de timezone. Basé sur Luxon. Utile pour les expirations, fenêtres de temps.
📤
Respond to Webhook
Renvoie une réponse HTTP au caller du webhook. Obligatoire pour les webhooks synchrones (quand l'appelant attend une réponse). Avec body, status code, headers.
🧩
Community Nodes : Settings → Community Nodes → Install. Des centaines de nodes créés par la communauté pour des apps non couvertes officiellement. Les plus utiles : n8n-nodes-browserless (scraping), n8n-nodes-pdf (manipulation PDF), n8n-nodes-openrouter (tous les LLMs via une API). Vérifier le nombre d'étoiles GitHub avant d'utiliser.

Expressions & Manipulation de données

Le système d'expressions de n8n permet d'accéder dynamiquement aux données de n'importe quel node précédent. C'est le "colle" qui lie tes nodes entre eux.

Syntaxe de base

Les expressions s'écrivent entre double accolades {{ expression }}. Tu peux les utiliser dans n'importe quel champ de n'importe quel node.

💡
Variables globales : $json = données de l'item courant. $input = input du node courant. $node["NomDuNode"] = données d'un node spécifique. $workflow = infos du workflow. $now = timestamp courant. $env = variables d'environnement.

Accéder aux données du node précédent

// $json — données de l'item courant (le plus utilisé) {{ $json.email }} → "[email protected]" {{ $json.user.name }} → "Alice" {{ $json["first-name"] }} → champs avec tirets {{ $json.tags[0] }} → premier élément d'un array {{ $json.items.length }} → taille d'un array // Accéder à un node précis par son nom {{ $node["HTTP Request"].json.id }} {{ $("Webhook").item.json.body.email }} // Infos contextuelles {{ $now.toISO() }} → "2025-02-19T14:30:00.000Z" {{ $workflow.id }} → ID du workflow {{ $execution.id }} → ID unique de l'exécution

Opérations sur les chaînes (String)

// Manipulation de texte {{ $json.name.toUpperCase() }} → "ALICE" {{ $json.name.toLowerCase() }} → "alice" {{ $json.name.trim() }} → sans espaces en début/fin {{ $json.description.slice(0, 100) }} → 100 premiers caractères {{ $json.text.replace("old", "new") }} {{ $json.email.split("@")[1] }} → domaine de l'email {{ $json.name.includes("Alice") }} → true/false // Template strings {{ `Bonjour ${$json.prenom}, votre commande ${$json.orderId} est prête` }} // Test si vide {{ $json.field ? $json.field : "valeur par défaut" }} {{ $json.field ?? "valeur par défaut" }} // Nullish coalescing

Opérations sur les nombres

{{ Math.round($json.amount * 1.2) }} → +20% arrondi {{ Math.max($json.a, $json.b) }} {{ ($json.done / $json.total * 100).toFixed(1) }} → "60.0" {{ Number($json.prix).toLocaleString("fr-FR") }} → "1 500"

Opérations sur les dates (Luxon)

// n8n utilise Luxon pour les dates {{ $now.toFormat("dd/MM/yyyy") }} → "19/02/2025" {{ $now.toFormat("dd MMMM yyyy", { locale: "fr" }) }} → "19 février 2025" {{ $now.plus({ days: 7 }).toISO() }} → dans 7 jours {{ $now.minus({ months: 1 }).startOf("month").toISO() }} {{ DateTime.fromISO($json.date).toFormat("dd/MM/yyyy") }} {{ $now.diff(DateTime.fromISO($json.created), "days").days }} → nb jours {{ $now.weekday }} → 1=lundi, 7=dimanche

Opérations sur les arrays et objets

// Arrays {{ $json.tags.join(", ") }} → "IA, Startup, Product" {{ $json.items.filter(i => i.active).length }} → nb items actifs {{ $json.items.map(i => i.name) }} → array de noms {{ $json.items.find(i => i.id === "abc") }} {{ $json.prices.reduce((sum, p) => sum + p, 0) }} → somme {{ JSON.stringify($json.metadata) }} → objet → string {{ JSON.parse($json.jsonString) }} → string → objet // Obtenir toutes les clés d'un objet {{ Object.keys($json.data) }} {{ Object.values($json.data) }}

Le node Code — JavaScript & Python full power

Code Node — JavaScript
// Accès aux items entrants
const items = $input.all();

// Transformer chaque item
const result = items.map(item => {
  const data = item.json;
  return {
    json: {
      id: data.id,
      fullName: `${data.firstName} ${data.lastName}`,
      emailDomain: data.email.split('@')[1],
      isHighValue: data.mrr > 500,
      processedAt: new Date().toISOString()
    }
  };
});

return result;

// ─── Exemple 2 : Appel fetch dans le Code node ───
const items = $input.all();
const results = [];

for (const item of items) {
  const response = await fetch(
    `https://api.clearbit.com/v2/people/email/${item.json.email}`,
    { headers: { 'Authorization': `Bearer ${$env.CLEARBIT_API_KEY}` } }
  );
  const enriched = await response.json();
  results.push({ json: { ...item.json, clearbit: enriched } });
}

return results;

Comment construire des workflows qui prennent des décisions, traitent des données en parallèle, bouclent, et fusionnent des résultats.

Exemples de conditions IF

Gestion des erreurs

Un workflow sans gestion d'erreurs est un workflow qui va silencieusement casser en production. Voici comment construire des automations robustes.

Les 3 niveaux de gestion d'erreur

🔧
Node-level
Settings de chaque node : "Continue On Fail". Le workflow continue même si ce node échoue. Les items échoués vont vers la sortie "error" du node.
🔄
Retry
Sur chaque node, active "Retry On Fail" avec N tentatives et un délai entre chaque. Parfait pour les APIs instables (503, timeout).
🚨
Error Workflow
Workflow dédié déclenché par l'Error Trigger quand n'importe quel workflow échoue. Centralise toute la logique d'alerte et de logging.

Configurer "Continue On Fail"

Dans les Settings de chaque node → "On Error" → "Continue (using error output)". Le node ajoute maintenant un port rouge "error" en plus du port vert "success". Les items qui échouent partent dans le port rouge, les autres continuent normalement.

# ─── Workflow : "_Error Handler" ───

Retry automatique avec backoff exponentiel

Pattern : retry avec backoff dans Code Node
const maxRetries = 3;
let attempt = 0;
let lastError;

while (attempt < maxRetries) {
  try {
    const response = await fetch('https://api.example.com/data', {
      headers: { 'Authorization': `Bearer ${$env.API_KEY}` }
    });

    if (!response.ok) throw new Error(`HTTP ${response.status}`);

    const data = await response.json();
    return [{ json: data }];

  } catch (error) {
    lastError = error;
    attempt++;
    if (attempt < maxRetries) {
      // Backoff exponentiel : 1s, 2s, 4s
      await new Promise(r => setTimeout(r, 1000 * Math.pow(2, attempt - 1)));
    }
  }
}

throw new Error(`Failed after ${maxRetries} retries: ${lastError.message}`);
🚨
Erreurs silencieuses : Le danger n°1 en n8n. Si un node ne reçoit aucun item (ex: une query SQL vide), le workflow s'arrête proprement — sans erreur. Pour détecter ça, utilise IF → is Empty → Stop and Error après chaque requête critique. Ou utilise "Always Output Data" dans les settings du node.

Valider la signature d'un webhook Stripe

Code Node — vérification signature HMAC
const crypto = require('crypto');

const payload = $input.first().json.body;
const sig = $input.first().json.headers['stripe-signature'];
const secret = $env.STRIPE_WEBHOOK_SECRET;

// Extraire le timestamp de la signature
const parts = sig.split(',');
const timestamp = parts.find(p => p.startsWith('t=')).split('=')[1];
const v1 = parts.find(p => p.startsWith('v1=')).split('=')[1];

// Recalculer la signature
const signed = `${timestamp}.${JSON.stringify(payload)}`;
const expected = crypto.createHmac('sha256', secret).update(signed).digest('hex');

if (expected !== v1) throw new Error('Invalid Stripe signature');

return [{ json: payload }]; // Signature valide, continuer

RAG & Vector Stores

Donner une mémoire documentaire à ton agent IA. RAG = Retrieval Augmented Generation : l'agent cherche dans tes documents avant de répondre.

Architecture RAG en n8n

# ─── PHASE 1 : Indexation (run une seule fois) ─── Source de données (Google Drive, Notion, PDF, URLs) │ ▼ Document Loader (charge le contenu) │ ▼ Text Splitter (découpe en chunks de ~500 tokens) │ ▼ Embeddings Model (OpenAI text-embedding-3-small) │ ▼ Vector Store (stocke les vecteurs) │ Pinecone / Supabase pgvector / Qdrant / In-memory # ─── PHASE 2 : Requête (à chaque question) ─── Question utilisateur │ ▼ Embeddings (vectoriser la question) │ ▼ Vector Search (top 5 chunks les plus similaires) │ ▼ LLM + Contexte ("Réponds à cette question : ... en te basant sur : ...") │ ▼ Réponse avec sources

Workflow d'indexation complet

Pipeline RAG — indexation PDF + Supabase
# 1. Trigger : nouveau fichier dans Google Drive
Google Drive Trigger  (nouveau fichier dans dossier "docs")

# 2. Télécharger le fichier
Google Drive: Download File
  fileId: {{ $json.id }}

# 3. Charger le document
Default Data Loader
  Data Type: Binary
  Loader: PDF

# 4. Découper en chunks
Recursive Character Text Splitter
  Chunk Size: 500     # tokens par chunk
  Chunk Overlap: 50   # overlap entre chunks pour garder le contexte

# 5. Créer les embeddings + stocker dans Supabase
Supabase Vector Store  (mode: Insert Documents)
  Table: documents
  Embeddings: OpenAI Embeddings (text-embedding-3-small)
  Metadata: {
    source: {{ $json.name }},
    url: {{ $json.webViewLink }},
    indexed_at: {{ $now.toISO() }}
  }

🥇 Recette 1 — Pipeline de qualification de leads IA

Chaque nouveau lead depuis un formulaire est enrichi, scoré par un LLM, et routé automatiquement.

Lead Qualification Pipeline
# TRIGGER
Webhook  ← Tally form submission
  POST /webhook/lead-qualification
  Body: { name, email, company, message, budget }

# 1. Enrichissement via Apollo
HTTP Request  → Apollo API
  GET https://api.apollo.io/v1/people/match?email={{ $json.email }}
  Headers: Api-Key: {{ $env.APOLLO_KEY }}
→ Ajoute : title, seniority, linkedin_url, company_size, industry

# 2. Validation email via Hunter
HTTP Request  → Hunter.io
  GET https://api.hunter.io/v2/email-verifier?email={{ $json.email }}
→ score > 70 ? continuer : logger spam + stop

# 3. Scoring IA avec GPT-4o
OpenAI: Message a Model
  Model: gpt-4o-mini  # moins cher, suffisant pour scoring
  Prompt:
  "Analyse ce lead et retourne un JSON :
   { score: 0-100, tier: 'hot|warm|cold', reason: string, suggested_action: string }

   Lead data:
   - Company: {{ $json.company }} ({{ $json.company_size }} employees)
   - Title: {{ $json.title }}, {{ $json.seniority }}
   - Industry: {{ $json.industry }}
   - Budget: {{ $json.budget }}
   - Message: {{ $json.message }}

   Scoring criteria:
   - Decision maker (C-level/VP/Director) = +30pts
   - Company size 50-500 = +20pts
   - Budget > 10k = +25pts
   - Clear pain point in message = +15pts"

# 4. Parser le JSON de la réponse LLM
Code Node
  const parsed = JSON.parse({{ $json.choices[0].message.content }});
  return [{ json: { ...previousData, ai: parsed } }];

# 5. Routage selon le score
Switch
  Condition 1: score >= 80  → "Hot lead"
  Condition 2: score >= 50  → "Warm lead"
  Default:                  → "Cold lead"

# Branche HOT (score >= 80)
HubSpot: Create Deal  (pipeline: Enterprise, stage: SQL)
  + Slack: notify #sales-hot-leads
    "🔥 HOT LEAD: {{ name }} ({{ company }}) — Score: {{ score }}
     Raison: {{ reason }}
     LinkedIn: {{ linkedin_url }}"
  + Calendly: Create link  (ou Send email avec CTA calendrier)

# Branche WARM (score 50-79)
HubSpot: Create Contact  (lifecycle: Lead)
  + Resend: Send email  (séquence nurturing automatique)
  + Supabase: Log  dans table leads

# Branche COLD
Supabase: Insert  dans table cold_leads
  + Resend: email de remerciement générique

🥉 Recette 3 — Monitoring & alertes automatiques

Uptime + Performance Monitor
# TRIGGER : toutes les 5 minutes

  → Supabase: log incident
  → Wait: 10 min (éviter spam d'alertes)

# Monitoring daily report (schedule 9h tous les jours)

  → Supabase: query uptime stats last 24h
