Products & Prices — La séparation fondamentale

Un Product = ce que vous vendez (immuable). Un Price = comment vous le tarifez (peut changer). Un produit peut avoir plusieurs prix : mensuel, annuel, lifetime, promotionnel. Cette séparation permet de créer de nouveaux pricing sans toucher à l'objet métier.

typescript — Créer product + prices
import Stripe from 'stripe'
const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!)

// 1. Créer le produit
const product = await stripe.products.create({
  name: 'Pro Plan',
  description: 'Accès complet à toutes les features',
  metadata: { tier: 'pro', max_seats: '10' }
})

// 2. Prix mensuel flat
const monthlyPrice = await stripe.prices.create({
  product: product.id,
  unit_amount: 4900,          // 49€ en centimes
  currency: 'eur',
  recurring: { interval: 'month' },
  metadata: { billing_period: 'monthly' }
})

// 3. Prix annuel avec remise
const annualPrice = await stripe.prices.create({
  product: product.id,
  unit_amount: 47000,         // 470€/an (20% off)
  currency: 'eur',
  recurring: { interval: 'year' },
  metadata: { billing_period: 'annual', savings: '118' }
})

// 4. Prix per-seat (usage metered)
const seatPrice = await stripe.prices.create({
  product: product.id,
  unit_amount: 1500,          // 15€ par siège
  currency: 'eur',
  recurring: {
    interval: 'month',
    usage_type: 'licensed'    // quantité fixée au billing
  }
})

Upgrade / Downgrade — Proration

Quand un client change de plan en milieu de cycle, Stripe calcule automatiquement la différence au prorata. Vous choisissez le comportement.

typescript — Upgrade avec proration
// Récupérer l'abonnement actuel
const subscription = await stripe.subscriptions.retrieve(subscriptionId)

// Preview de la proration avant d'appliquer
const proration_date = Math.floor(Date.now() / 1000)

const invoice = await stripe.invoices.retrieveUpcoming({
  customer: subscription.customer as string,
  subscription: subscriptionId,
  subscription_items: [{
    id: subscription.items.data[0].id,
    price: newPriceId,
  }],
  subscription_proration_date: proration_date,
})

// Le montant dû immédiatement
console.log('Proration amount:', invoice.amount_due / 100, '€')

// Appliquer l'upgrade
const updated = await stripe.subscriptions.update(subscriptionId, {
  items: [{
    id: subscription.items.data[0].id,
    price: newPriceId,
  }],
  proration_behavior: 'create_prorations',  // ou 'none' ou 'always_invoice'
  proration_date,
})

// proration_behavior options :
// 'create_prorations'  → crédite/débite au prochain cycle
// 'always_invoice'     → facture immédiatement la différence
// 'none'               → aucune proration (prix plein au prochain cycle)

typescript — Annulation soft vs hard

  cancel_at_period_end: false,

Handler webhook production — Architecture complète

typescript — app/api/webhooks/stripe/route.ts
import { headers } from 'next/headers'
import Stripe from 'stripe'

export async function POST(req: Request) {
  const body = await req.text()   // text(), pas json() !
  const sig = headers().get('stripe-signature')!

  let event: Stripe.Event

  try {
    // 1. Vérifier la signature TOUJOURS en premier
    event = stripe.webhooks.constructEvent(
      body, sig,
      process.env.STRIPE_WEBHOOK_SECRET!
    )
  } catch (err) {
    console.error('Webhook signature invalid', err)
    return new Response('Unauthorized', { status: 400 })
  }

  // 2. Idempotence — éviter le double traitement
  const processed = await isEventAlreadyProcessed(event.id)
  if (processed) {
    return new Response('Already processed', { status: 200 })
  }

  // 3. Marquer comme "en cours" (avant traitement)
  await markEventAsProcessing(event.id)

  try {
    // 4. Router vers le bon handler
    switch (event.type) {
      case 'checkout.session.completed':
        await handleCheckoutCompleted(event.data.object)
        break
      case 'customer.subscription.updated':
        await handleSubscriptionUpdated(event.data.object)
        break
      case 'customer.subscription.deleted':
        await handleSubscriptionDeleted(event.data.object)
        break
      case 'invoice.payment_failed':
        await handlePaymentFailed(event.data.object)
        break
      case 'invoice.payment_succeeded':
        await handlePaymentSucceeded(event.data.object)
        break
      default:
        // Log les events non gérés pour future implémentation
        console.log(`Unhandled event: ${event.type}`)
    }

    // 5. Marquer comme traité avec succès
    await markEventAsDone(event.id)
    return new Response('OK', { status: 200 })

  } catch (err) {
    // 6. En cas d'erreur : laisser Stripe retenter
    // Stripe retente 3 fois sur 3 jours si non-200
    console.error(`Handler error for ${event.type}:`, err)
    await markEventAsFailed(event.id, err)
    return new Response('Internal Error', { status: 500 })
  }
}

CREATE TABLE stripe_webhook_events (
  id            TEXT        PRIMARY KEY,  -- event.id Stripe (evt_xxx)
  type          TEXT        NOT NULL,
  status        TEXT        NOT NULL      -- 'processing' | 'done' | 'failed'
                             DEFAULT 'processing',
  payload       JSONB,
  error_message TEXT,
  processed_at  TIMESTAMPTZ,
  created_at    TIMESTAMPTZ DEFAULT now()
);

-- Index pour les lookups fréquents
CREATE INDEX idx_webhook_status ON stripe_webhook_events(status);
CREATE INDEX idx_webhook_type   ON stripe_webhook_events(type);

-- Purger après 90 jours (rétention Stripe)
CREATE POLICY "retention" ON stripe_webhook_events
  FOR DELETE USING (created_at < now() - interval '90 days');

Stripe retente sur 3 jours : si votre endpoint répond avec une erreur 5xx ou un timeout, Stripe retente automatiquement avec backoff exponentiel pendant 3 jours. Handler dunning — Déclencher la séquence

async function handlePaymentFailed(invoice: Stripe.Invoice) {
  const customerId = invoice.customer as string

  // Récupérer l'user depuis votre DB
  const user = await db.users.findByStripeCustomerId(customerId)
  if (!user) throw new Error(`No user for customer ${customerId}`)

  // Compter les tentatives pour personnaliser le message
  const attemptCount = invoice.attempt_count  // 1, 2, 3...

  // Générer le lien de mise à jour de paiement
  const portalSession = await stripe.billingPortal.sessions.create({
    customer: customerId,
    return_url: `${APP_URL}/dashboard`,
    flow_data: { type: 'payment_method_update' }
  })

  // Router vers le bon template selon l'attempt
  const templates = {
    1: 'dunning_first_attempt',
    2: 'dunning_second_attempt',
    3: 'dunning_final_warning',
  }

  await email.send({
    to: user.email,
    template: templates[attemptCount] ?? 'dunning_final_warning',
    variables: {
      firstName:    user.firstName,
      updateUrl:    portalSession.url,
      invoiceUrl:   invoice.hosted_invoice_url,
      amount:       (invoice.amount_due / 100).toFixed(2),
      attemptCount,
      daysRemaining: 14 - (attemptCount * 3),
    }
  })

  // Log pour analytics
  analytics.track('payment_failed', {
    userId: user.id, attemptCount,
    amount: invoice.amount_due,
    invoiceId: invoice.id
  })
}

Recognition

Distinguer cash collecté et revenu reconnu. Un paiement annuel de 1 200€ en janvier n'est pas 1 200€ de revenu en janvier — c'est 100€/mois sur 12 mois. IFRS 15 l'impose, Stripe Revenue Recognition l'automatise.

§5
IFRS 15
Compliant
MRR ARR Deferred Revenue IFRS 15 Accrual Accounting Stripe Revenue Recognition

Le principe — Cash vs Revenu reconnu

Paiement annuel 1 200€ reçu le 1er janvier

Cash collected (bilan) : +1 200€ le 1er janvier
Revenue recognized (P&L) : +100€/mois sur 12 mois

Deferred revenue (dette) : −1 100€ en janvier
−1 000€ en février (−100€ reconnus)
0€ au 31 décembre

Le deferred revenue est une dette envers le client — vous avez encaissé mais pas encore livré le service. Cette dette disparaît au fur et à mesure que le service est rendu.

MRR, ARR, et les métriques qui comptent

MétriqueDéfinitionCalcul StripeUsage
MRR Monthly Recurring Revenue Σ (montant_mensuel des subscriptions actives) Santé mensuelle
ARR Annual Recurring Revenue MRR × 12 Valorisation, fundraising
New MRR MRR ajouté ce mois Nouvelles subscriptions actives ce mois Performance acquisition
Expansion MRR MRR additionnel d'existants Upgrades + add-ons ce mois Land & expand
Churned MRR MRR perdu ce mois Subscriptions annulées × montant mensuel Rétention
Net Revenue Churn Churned − Expansion (Churned MRR − Expansion MRR) / MRR début mois Santé NRR

// Calculer le MRR depuis les subscriptions actives
export async function calculateMRR(): Promise<number> {
  let mrr = 0
  let hasMore = true
  let startingAfter: string | undefined

  while (hasMore) {
    const subs = await stripe.subscriptions.list({
      status: 'active',
      limit: 100,
      expand: ['data.items.data.price'],
      starting_after: startingAfter,
    })

    for (const sub of subs.data) {
      for (const item of sub.items.data) {
        const price = item.price
        if (!price.unit_amount) continue

        const qty = item.quantity ?? 1
        const monthly = price.recurring?.interval === 'year'
          ? (price.unit_amount * qty) / 12
          : price.unit_amount * qty

        mrr += monthly
      }
    }

    hasMore = subs.has_more
    startingAfter = subs.data.at(-1)?.id
  }

  return mrr / 100   // convertir centimes en euros
}

Stripe Sigma — SQL sur vos données de revenu

sql — Stripe Sigma — MRR par mois
-- MRR mensuel sur les 12 derniers mois
-- À exécuter dans Stripe Dashboard → Sigma

SELECT
  date_trunc('month', created) AS month,
  SUM(amount_paid) / 100.0     AS cash_collected,
  COUNT(DISTINCT subscription)  AS active_subscriptions,
  AVG(amount_paid) / 100.0     AS avg_invoice_value
FROM invoices
WHERE
  status = 'paid'
  AND created >= now() - interval '12 months'
  AND subscription IS NOT NULL   -- exclure one-time payments
GROUP BY 1
ORDER BY 1 DESC;

-- Churn MRR ce mois
SELECT
  SUM(plan_amount) / 100.0 AS churned_mrr,
  COUNT(*) AS churned_subscriptions
FROM subscriptions
WHERE
  status = 'canceled'
  AND canceled_at >= date_trunc('month', now());
◈
NRR (Net Revenue Retention) — La métrique de maturité : un NRR > 100% signifie que votre base existante croît d'elle-même par expansion, compensant le churn. C'est le signal que vous avez un vrai product-market fit et un pricing qui s'adapte à la valeur perçue croissante. Ciblez >110% pour une valorisation SaaS optimale.

Stripe Radar — Prévention de la fraude

Radar est le moteur anti-fraude de Stripe, entraîné sur l'ensemble du réseau Stripe. Il bloque ~99% des fraudes automatiquement. Configurez des règles custom pour votre secteur.

Règles Radar — Stripe Dashboard
// Règles custom à configurer dans Dashboard → Radar → Rules
// Syntaxe Radar Rule Language

// Bloquer les paiements de certains pays à risque élevé
block if :ip_country: in ('XX', 'YY')

// Demander 3DS pour les montants élevés
request_three_d_secure if :amount_in_eur: > 500

// Review manuelle pour les nouvelles cartes sur des montants importants
review if :card_funding: = 'prepaid'
review if :is_new_card: = true and :amount_in_eur: > 200

// Bloquer les VPN/Tor connus
block if :ip_is_anonymous_proxy: = true

// Metadata custom dans Payment Intent pour les règles
const paymentIntent = await stripe.paymentIntents.create({
  amount, currency,
  metadata: {
    user_id:       userId,
    account_age:   daysSinceSignup.toString(),
    is_verified:   isEmailVerified.toString(),
  }
})
