Module 01

GitHub

La plateforme où vit tout le code du monde. Comprendre GitHub, c'est comprendre comment les équipes tech collaborent et livrent des produits.

L'analogie parfaite

Imagine que ton code est un document Google Docs. Git = l'historique des versions. GitHub = Google Drive, là où tout le monde peut accéder au doc, proposer des modifications, et voir qui a changé quoi et quand.

Git vs GitHub — la confusion #1

⚙️
Git
Un logiciel installé sur ton ordinateur. Il track les changements de ton code localement. Créé par Linus Torvalds (le créateur de Linux) en 2005. Gratuit et open-source. C'est le moteur.
LocalLogicielOpen Source
🐙
GitHub
Un site web (github.com) qui héberge tes repos Git dans le cloud. Appartient à Microsoft ($7.5B en 2018). 100M+ développeurs. C'est la carrosserie — la partie visible et sociale.
MicrosoftCloudSocial
Alternatives à GitHub

GitLab : Concurrent, très populaire en entreprise et open-source. Meilleur pour le DevOps intégré. Bitbucket : Fait par Atlassian (Jira), intégration parfaite avec leurs outils. Mais GitHub reste le standard avec 90% des projets open-source.

Les concepts Git fondamentaux

01
Repository (Repo)
Le dossier de ton projet versionné. Contient tout ton code + l'historique complet de toutes les modifications jamais faites.
02
Commit
Une "photo" de ton code à un instant T. Chaque commit a un message ("Fix login bug") et un identifiant unique. L'historique = la somme de tous les commits.
03
Branch
Une copie parallèle du code. Tu crées une branche pour développer une feature sans casser le code principal (main). Une fois fini, tu "merge" dans main.
04
Pull Request (PR)
Demande de merger ta branche dans main. Les collègues reviews ton code, laissent des commentaires. Acceptée = le code est mergé. C'est ainsi que les équipes collaborent.
05
Fork
Copier le repo de quelqu'un d'autre dans ton compte GitHub. Tu peux modifier librement sans affecter l'original. Base de tout l'open-source.
06
Clone
Télécharger un repo GitHub sur ta machine locale. Après le clone, tu as tout le code + l'historique en local et tu peux travailler offline.

Les commandes Git essentielles — pratique

terminal
# 1. Initialiser un projet git init # Créer un repo dans le dossier actuel # 2. Connecter à GitHub git remote add origin https://github.com/toi/mon-projet.git # 3. Le workflow quotidien git status # Voir ce qui a changé git add . # "Stager" tous les fichiers modifiés git commit -m "Ajout page login" # Créer un commit avec un message git push origin main # Envoyer sur GitHub # 4. Récupérer les changements des collègues git pull origin main # Mettre à jour ton code local # 5. Merge une branche git merge feature/nouvelle-page # Intégrer les changements dans main
GitHub Actions — l'automatisation

GitHub Actions permet d'automatiser des tâches à chaque push : lancer des tests automatiques, déployer sur Vercel, envoyer une notif Slack. C'est le CI/CD intégré à GitHub. Quand ton ingénieur dit "la pipeline est cassée", il parle de ça. Gratuit jusqu'à 2000 minutes/mois.

GitHub dans la vraie vie d'une startup

1
Le repo principal
Tout le code de la startup vit dans 1 ou plusieurs repos GitHub. Privés en général (pas visibles publiquement). L'accès se gère par équipe : devs ont accès au code, pas les autres.
2
Branch protection
La branche "main" est protégée — personne ne peut pousser directement sans une PR reviewée. Évite que quelqu'un casse la prod par accident.
3
Issues & Projects
GitHub a un outil de gestion de tickets (Issues) et un Kanban (Projects). Certaines startups gèrent tout là. D'autres préfèrent Linear ou Jira — mais GitHub est souvent suffisant en early stage.
4
GitHub Copilot
L'IA de coding de GitHub (Microsoft + OpenAI). Complète ton code en temps réel. $10/mois. Augmente la productivité des devs de 30-55% selon des études. Révolutionnaire pour les juniors.
Module 02

Vercel & Netlify

Les plateformes qui transforment ton code GitHub en un site live accessible partout dans le monde — en quelques secondes, sans gérer de serveurs.

L'analogie parfaite

Vercel, c'est comme un imprimeur ultra-rapide. Tu écris ton manuscrit (code), tu le donnes à Vercel, et en 30 secondes il est imprimé (déployé) et disponible dans toutes les librairies du monde (CDN global). Tu n'as jamais touché à la presse.

Le problème qu'ils résolvent

Avant Vercel/Netlify, déployer un site web voulait dire : louer un serveur, l'installer, configurer Linux, Nginx, les certificats SSL, le DNS, les mises à jour de sécurité… Une journée de travail minimum. Maintenant : 30 secondes.

▲
Vercel
Créé par les auteurs de Next.js. La plateforme de référence pour React/Next. Déploiement instantané depuis GitHub. Preview URLs pour chaque PR. Analytics intégrés. Fondée en 2015, valorisée $3.25B.
Next.js natifFreemium
🦊
Netlify
Pionnière du "JAMstack". Plus polyvalente que Vercel (tous frameworks). Netlify Drop = déploiement par drag & drop. Forms, Identity, Edge Functions intégrés. Très populaire pour les sites statiques.
PolyvalentFreemium
Vercel vs Netlify — lequel choisir ?

Tu fais du Next.js ? → Vercel sans hésiter (même équipe, intégration parfaite). Tu fais autre chose (React simple, Vue, Hugo, Astro) ? → Les deux sont équivalents. Pour débuter ? Netlify Drop est le plus simple (glisser-déposer). Pour une vraie startup → Vercel.

Preview Deployments — la feature killer

Chaque Pull Request génère automatiquement une URL de preview unique. Exemple : ma-startup-pr-42.vercel.app. Tu peux la partager avec ton designer ou tes investisseurs pour voir les changements avant de les merger en prod. C'est une des features les plus utilisées par les équipes product.

Déploiement pratique — étape par étape

1
Connecte GitHub à Vercel
Va sur vercel.com → "New Project" → "Import Git Repository" → Autorise Vercel à accéder à ton GitHub. Sélectionne le repo.
2
Vercel détecte ton framework
Vercel reconnaît automatiquement Next.js, React, Vue, etc. et configure la build. En général zero config nécessaire.
3
Variables d'environnement
Tes clés secrètes (API keys, BDD passwords) vont dans "Environment Variables" dans Vercel — jamais dans ton code GitHub. C'est là que tu mets OPENAI_API_KEY=sk-...
4
Domaine custom
Dans les settings Vercel → Domains → ajoute ton domaine (ex: masaas.com). Vercel configure le SSL (HTTPS) automatiquement en quelques minutes.
vercel.json — config optionnelle
// Fichier optionnel à la racine de ton projet { "framework": "nextjs", "buildCommand": "npm run build", "outputDirectory": ".next", "env": { "NEXT_PUBLIC_API_URL": "@api_url" // Référence une env var Vercel }, "rewrites": [ { "source": "/api/:path*", "destination": "/api/:path*" } ] }

Module 03

AWS · GCP · Azure

Les trois géants du cloud qui font tourner 90% d'internet. Comprendre ce qu'ils vendent et quand les utiliser.

L'analogie parfaite

Le cloud, c'est comme louer de l'électricité au lieu d'avoir votre propre générateur. Avant le cloud, les entreprises achetaient des serveurs physiques. Aujourd'hui elles louent de la puissance de calcul à la demande, au centime près.

Pourquoi le cloud a tout changé

Avant AWS (2006), lancer une startup nécessitait d'acheter des serveurs ($10k-100k+), louer un datacenter, embaucher des sysadmins. Cela prenait des mois. AWS a rendu ça possible en quelques clics pour $0 au départ. Netflix, Airbnb, Uber ont tous commencé sur AWS. C'est la démocratisation totale de l'infrastructure tech.

Amazon AWS
  • Lancé en 2006, le pionnier
  • ~31% du marché cloud
  • 200+ services disponibles
  • S3 (stockage), EC2 (serveurs), Lambda (serverless)
  • Le plus mature et le plus utilisé
  • Documentation immense mais complexe
  • Free tier 12 mois très généreux
Google GCP
  • Google Cloud Platform
  • ~11% du marché cloud
  • Meilleur pour les données & IA/ML
  • BigQuery (analytics), TPUs (IA), Vertex AI
  • Firebase = backend as a service pour apps mobiles
  • Interface plus simple qu'AWS
  • $300 de crédits à l'inscription
Microsoft Azure
  • ~24% du marché cloud
  • Dominant dans les grandes entreprises
  • Intégration native Microsoft (Office 365, Active Directory)
  • Azure OpenAI Service = GPT-4 en enterprise
  • Certifications Azure très reconnues
  • Complexe pour les débutants
  • $200 de crédits à l'inscription
Lequel choisir pour une startup ?

AWS si tu veux le maximum de ressources, tutorials, et qu'aucun autre critère ne s'impose. C'est le plus safe. GCP si ton produit est centré IA/ML ou data analytics. Azure si tu vends à des grandes entreprises qui sont déjà dans l'écosystème Microsoft. En early stage, commence par AWS.

Les services cloud essentiels à connaître

🖥️
Compute — les serveurs virtuels
AWS EC2 / GCP Compute Engine / Azure VMs. Tu loues un "ordinateur" dans le cloud. Tu choisis la taille (CPU, RAM). Tu paies à l'heure. C'est là que tourne ton backend.
EC2VMs$0.01-10/heure
🗄️
Storage — le stockage
AWS S3 (Simple Storage Service) = le Google Drive des devs. Stocke fichiers, images, vidéos, backups. Utilisé par Netflix, Airbnb, Slack. ~$0.023/GB/mois. Le service cloud le plus utilisé au monde.
S3Blob Storage
⚡
Serverless — Functions
AWS Lambda / GCP Cloud Functions / Azure Functions. Exécute du code sans gérer de serveur. Tu paies uniquement quand ta fonction tourne. Parfait pour les APIs légères et les tâches automatisées.
LambdaPay per use
🗃️
Managed Databases
AWS RDS (Postgres/MySQL managé), DynamoDB (NoSQL), Firebase Firestore (GCP). La base de données tourne dans le cloud, backups automatiques, scaling auto. Tu ne gères pas le serveur.
RDSDynamoDB
🌐
CDN — Content Delivery Network
AWS CloudFront. Distribue ton contenu (images, videos, pages) depuis des serveurs proches de l'utilisateur. Quelqu'un à Tokyo reçoit le contenu depuis un serveur japonais, pas depuis Paris. Vercel fait ça automatiquement.
CloudFrontPerformance
🤖
AI / ML Services
AWS Bedrock (accès à Claude, Llama, etc.), GCP Vertex AI, Azure OpenAI. Les clouds exposent les grands modèles IA via API avec gestion enterprise (sécurité, compliance, SLA).
BedrockVertex AI

Pratique — lancer ton premier serveur AWS

1
Créer un compte AWS Free Tier
aws.amazon.com → "Créer un compte gratuit". CB requise mais ne sera pas débitée si tu restes dans les limites. Free tier = 12 mois d'accès gratuit à la plupart des services basiques.
2
AWS Console — le tableau de bord
Console.aws.amazon.com — L'interface principale. 200+ services listés. Ne panique pas. Tu n'en utiliseras que 5-10 au début. Utilise la barre de recherche pour trouver ce dont tu as besoin.
3
Créer un bucket S3 (stockage)
S3 → "Create Bucket" → Nom unique mondial → Région (eu-west-1 = Irlande pour l'Europe) → Upload tes fichiers. Free tier : 5GB gratuits la première année.
4
Attention aux coûts !
Configure des billing alerts immédiatement (AWS Budgets → Create Budget → $10/mois). Des erreurs de config peuvent générer des factures surprises. C'est l'erreur classique de tout développeur débutant sur AWS.
Railway & Render — les alternatives simples

Si AWS/GCP/Azure te semblent trop complexes pour démarrer, Railway.app et Render.com sont des alternatives parfaites pour les startups early stage. Tu déploies ton backend comme tu déploies sur Vercel — en 2 clics depuis GitHub. Moins puissants mais infiniment plus simples. Parfait jusqu'à quelques milliers d'utilisateurs.

Scénario réel — tu codes une feature

1
Tu crées une branche sur GitHub
git checkout -b feature/page-pricing — Tu travailles sur ta feature sans toucher au code de prod.
2
Tu pushs — Vercel crée une preview
git push origin feature/page-pricing → Vercel détecte le push et déploie automatiquement sur une URL de preview. Tu vois ta feature live en 30 secondes.
3
Tu ouvres une Pull Request
Sur GitHub, tu ouvres une PR. Ton co-founder review le code, laisse des commentaires. Les GitHub Actions lancent les tests automatiques. Tout est vert → la PR est approuvée.
4
Merge → Déploiement en prod
Tu merges la PR dans main. Vercel détecte le merge sur main et redéploie automatiquement en production. L'URL masaas.com est mise à jour. Zero downtime.
5
Le backend tourne sur Railway/AWS
Ton API FastAPI tourne sur Railway (auto-déployée depuis le même GitHub). S3 stocke les uploads users. Supabase gère la base de données. Tout est connecté via variables d'environnement.
